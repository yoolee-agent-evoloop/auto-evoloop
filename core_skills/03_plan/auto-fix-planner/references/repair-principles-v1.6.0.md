## 一、修复策略优先级

| 优先级 | 策略 | 说明 |
|--------|------|------|
| **高** | 工具/架构修复 | 计算、查询、格式等问题优先从工具视角解决 |
| **高** | 补充知识库/数据 | 先确保数据完整，再考虑逻辑约束 |
| **高** | 编排层/路由修复 | 已有能力但未利用 → 修复路由，不修复内容 |
| **高** | 优化 Skill / Skill description | 增强现有 Skill 能力或改善触发条件 |
| **中** | 纯逻辑原则声明 | 用抽象因果逻辑概括行为，禁止 few-shot 枚举 |
| **中** | 架构层调整 | 结构化 memory、状态管理优化 |
| **低** | Prompt 约束堆砌 | 在 prompt 层面增加约束/强调 |
| **禁止** | Few-shot 示例 | 默认禁止；仅当纯逻辑原则确认不可行且标注 `[Few-shot 兜底]` 时例外 |
| **禁止** | 强调/加粗/重复声明 | 不符合最佳实践 |

## 二、修复内容生成硬约束

### 2.1 Anti-Few-shot 规则

修复内容中严禁以下模式：
- `（如"X"、"Y"、"Z"等）` 形式的举例枚举
- `禁止使用"X"..."Y"...等表达` 形式的具体话术列举
- `示例：用户说"X" → 回复"Y"` 形式的输入输出示例对

**替代策略层级**（严格按顺序）：
1. **纯逻辑原则**：用抽象因果推理描述行为规则
2. **COT 推理引导**：用思维链引导 agent 自行推导正确行为
3. **最后手段——受控 Few-shot**：标注 `[Few-shot 兜底]` 并说明理由

### 2.2 Agent 能力边界

- **Master Agent**：仅路由和整合，不直接回复/内容生成
- **子 Agent**：其 skill 范围内的内容生成
- **工具**：特定功能执行，无决策能力

修复方案不得要求 agent 执行架构角色之外的功能。

### 2.3 路由/编排优先

预检：现有架构中是否已有可胜任的子 Agent？
- 有 → Step 1 必须修复路由/编排
- 无 → 才考虑 Skill 内容修复或新建 Skill

### 2.4 知识/数据优先

1. 先检查知识库/工具返回数据是否缺失 → 缺失则 Step 1 补充数据
2. 数据完整但未正确使用 → 添加推理逻辑/原则约束
3. 逻辑约束不生效 → 更具体的描述（需有证据）

### 2.5 工具优先

以下问题必须首先考虑工具侧解决：
- 计算类 → 计算器工具
- 数据查询类 → 数据库/API 工具
- 格式转换类 → 格式化工具

### 2.6 Step 1 充分性

Step 1 必须为独立充分方案——仅执行 Step 1 即应能解决或显著缓解核心问题。Step 2-3 为备选/加固。

### 2.7 新 Skill 创建门槛

同时满足以下条件才建议新建 Skill：
1. 高频/高普适性
2. ROI 合理
3. 能力边界清晰

无法判断频次时标注 `[建议下游汇总时评估频次]`。

## 三、优先级评分公式

```
Priority Score = 杠杆率 × 成本分 × 确定性分
```

| 维度 | 定义 | 评分标准 |
|------|------|---------|
| **杠杆率** | 单次修复可能覆盖的根因数量 | 高(3)：覆盖>=2个根因；中(2)：覆盖1个；低(1)：仅缓解 |
| **成本分** | 修复的实施成本 | 低(3)：Skill/工具优化；中(2)：Prompt 原则声明；高(1)：架构改动 |
| **确定性分** | 修复效果的可验证程度 | 高(3)：可即时测试；中(2)：需部署后验证；低(1)：难以验证 |

### 问题解决完整度加权

- 若 Step 覆盖的根因全部是次要根因（非根因链起始节点），Priority Score × 0.5
- 若 Step 仅解决部分症状且不覆盖根因链头部 → 不排在 Step 1

**评判方法**：对每个 Step 问"如果只做这一步，问题能解决吗？"
- 能 → 候选 Step 1
- 不能但处理根因链头部 → 候选 Step 1
- 不能且处理根因链尾部 → 降级到 Step 2/3

## 四、评估器审视原则

在开始归因前审视评估器评分理由：
- 评估器指出的问题是否在 Trace 中可观测？
- 评估器的"期望输出"是否合理？
- 如果评估器评分理由与 Trace 实际行为矛盾 → `target_layer: PX_eval`

## 五、修复方案撰写检查清单

- [ ] **反补丁**：避免在 prompt 层面堆砌约束/强调
- [ ] **层级选择**：优先 Skill/工具层面
- [ ] **根本原因**：处理根本原因而非表面症状
- [ ] **杠杆点**：识别能连带解决其他问题的修复点
- [ ] **业务对齐**：是否需要业务确认
- [ ] **Anti-Few-shot**：After 文本无 few-shot 枚举
- [ ] **Agent 能力边界**：不违反架构角色
- [ ] **路由优先**：已有能力先修路由
- [ ] **知识优先**：先检查数据完整性
- [ ] **工具优先**：计算/查询优先工具方案
- [ ] **Step 1 充分性**：独立解决核心问题
- [ ] **新 Skill 门槛**：满足三重条件
- [ ] **评估器审视**：评分理由是否可靠
- [ ] **组件名准确性**：名称与 Harness 一致
