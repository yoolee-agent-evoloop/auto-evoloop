## 本 stage 角色
单条 case 的根因归因——trace + Harness → root cause + evidence chain + confidence + repair suggestion；HITL 2（采纳 / 修改 / 归并 / 拒绝）后产出 feedback.json。

对应 stage skill：`core_skills/02_analyze/auto-single-case-analyzer/SKILL.md`

## 输入 artifacts
- 当前 case 的 trace JSON / stderr / artifact 路径
- 单 case attribution 输出（schema 见 auto-single-case-analyzer/）
- HITL 2 的 feedback.json（verdict / revision / merge_group / reject_reason）
- 同 case 的多次 run 结果（如有，验证归因稳定性）
- reproduction params（seed / 模型版本 / 时间戳）

## 4 类根因在 S2 的典型 surface

> 指引而非清单——SKILL.md Step 2 的 4 类原理才是查的依据；这里只列"本 stage
> 最常出现在哪些 surface"，帮你定位。具体洞察由模型从原理推导出来。

- **A 测量信号** → trace / log / artifact 完整性 / 同 case 多 run 的归因一致性 / 上游 scorer 单点判定是否进入归因前提 / reproduction 是否真可复现 / HITL 2 的 reject 比例是否在合理区间（过高暗示归因质量塌方）
- **B 决策规则** → 归因 prompt / attribution schema 是否覆盖各种 root cause 形态 / root cause vs symptom vs 触发条件的层级是否清晰 / confidence 给定逻辑是否对应可观测证据 / merge_group 判定标准 / revise vs reject 边界
- **C 跨层信号** → attribution schema（`failure_type_primary` / `failure_type_secondary` / `target_layer` / `repair_directions[].location` / `repair_directions[].strategy`）是否对齐下游 fix-planner 消费字段 / evidence anchor（log 行号 / hash）多 run 间稳定可指 / feedback.json 中 `feedback[].verdict` / `feedback[].merge_group` / `feedback[].revision` 字段下游 fix-planner 是否真消费 / 归因 rationale 是 prose-only 还是结构化
- **D 成本曲线** → 同 case 反复归因次数 / 人工补正归因（revise）比例 / 跨轮归因结果是否反复 / 同根因跨 case 反复出现而 fix-planner 未沉淀 / HITL 2 累积耗时占比

## 数值一致性必查
- confidence 分数口径在多 run 间是否一致
- evidence anchor（log 行号 / hash）多 run 间稳定性
- HITL 2 的 reject 比例 / revise 比例 / merge 比例三档分布
- 相似 case 召回阈值（如有）

## v1 → v1+ 升级提示
跑过 ≥ 3 轮 S2 反思后，若发现某些 surface 反复命中或反复无产出，更新本文件。
