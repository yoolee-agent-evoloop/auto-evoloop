## 本 stage 角色

## 4 类根因在 S1 的典型 surface

> 指引而非清单——SKILL.md Step 2 的 4 类原理才是查的依据；这里只列"本 stage
> 最常出现在哪些 surface"，帮你定位。具体洞察由模型从原理推导出来。

- **A 测量信号** → trace JSON 完整性 / schema.json 列映射准确性 / Excel 解析鲁棒性（中文列头 / 合并单元格 / 空值）/ thread_id 抓取成功率 / 跨版本 trace 字段一致性 / 通过阈值 (passing_threshold) 是否真划得开通过 vs 失败
- **B 决策规则** → "什么算 badcase" 的阈值定义 / selected 默认值（全选 vs 部分）/ analysis_mode（A 单版本 vs B 多版本对比）选择规则 / Harness 拉取失败时的兜底（停 vs 用旧）
- **C 跨层信号** → analysis_manifest schema 是否对齐下游 single-case-analyzer 消费字段 / Harness.md 的 prompt / 工具定义 / 编排代码三段是否被下游真读 / selected 标签下游是否消费 / multi-version 场景的 `paired_trace_path` 字段（per auto-trace-prep schema）是否非空 / `analysis_mode` 与下游路径选择是否一致
- **D 成本曲线** → 同一批数据 trace 反复重抓次数 / HITL 1 反复确认次数（schema 不清晰征兆）/ 跨轮 manifest schema 漂移 / Harness 文件命名口径漂移 / 跨轮 selected 标注规则不一致

## 数值一致性必查
- passing_threshold 在跨轮间是否一致
- trace 抓取成功率（成功 trace 数 / 期望 trace 数）
- schema.json 中评分范围与 Excel 实际值范围的一致性
- 跨版本对比时 thread_id 配对率

## v1 → v1+ 升级提示
跑过 ≥ 3 轮 S1 反思后，若发现某些 surface 反复命中或反复无产出，更新本文件。
