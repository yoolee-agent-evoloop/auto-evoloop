# Meta-Reflection: S4 (execute) Module

> **Status**: v1 production-ready — 已 已验证轮次 实测验证，是 v1 唯一经过 calibration 的 stage 模块。

## 本 stage 角色
fix_plan → 实际 commit → eval → COMMIT/ROLLBACK gate → progressive_log 落库

主要决策点：D1（每 iter COMMIT/ROLLBACK）+ D2（CONTINUE/EXIT）+ EXIT 阶段的 PASS/REPLAN gate。

## 输入 artifacts（按 SKILL.md Step 2 的 4 类证据源映射）

> SKILL.md Step 2 的 4 类原理段给的是泛化"证据源类型"；下面是 S4 上每类
> 在本项目（biz_a evoloop）的实际 artifact 落点。**该映射是 stage-specific
> 的，不影响其他 stage 模块**。

**A 类（测量信号）**
- `{round_dir}/execution/eval_results/scored_run{1,2,3}.csv` — 多次运行结果（跨 run 算 scorer variance / noise floor）
- `{round_dir}/execution/eval_results/scored_*.csv` / `compare_*.json` — 测量工具输出
- `{round_dir}/execution/sample_manifest.json` — 抽样 manifest
- `scripts/run_eval/*` / `scripts/score/*` / `scripts/small_sample/*` — 工具链脚本（稳定性 / 重启冷启 / 输出格式一致性）

**B 类（决策规则）**
- `core_skills/04_execute/auto-fix-executor/SKILL.md` — D1 / D2 算法代码 + tolerance 公式
- `{round_dir}/fix_plan.md` — 规则参数（threshold / tolerance）+ 规则诞生依据（expected_tradeoffs / interaction_risks 假设）

**C 类（跨层信号）**
- `{round_dir}/fix_plan.md` — 来自 S3 的上游 artifact（特别看 expected_tradeoffs / interaction_risks / rollback prose）
- `{round_dir}/execution/progressive_log.json` — 本 stage 决策记录 + d1/d2 rationale 字段消费情况
- `{round_dir}/optimization_report.md` — 本 stage 输出（下游 R+1 fix_planner 读取）

**D 类（成本曲线）**
- `{round_dir}/../round_<N-1>/meta_reflection/S4.md`（如存在）— 上轮 P0 修了几个？哪些 fingerprint 重复出现？
- `{round_dir}/../round_<N-1>/execution/progressive_log.json`（如存在）— 上轮 vs 本轮 baseline / tolerance / 决策结构是否漂移
- 当前 conversation 中的 HITL 介入痕迹（人工停任务 / 手写 ad-hoc 脚本 / 手动同步状态字段 / 容忍 infra 噪音）
- git log 同字段（如 `stable_snapshot_commit`）的 reflog / 修改频次

## 4 类根因在 S4 的典型 surface

> 指引而非清单——SKILL.md Step 2 的 4 类原理才是查的依据；这里只列"本 stage
> 最常出现在哪些 surface"，帮你定位。具体洞察由模型从原理推导出来。

- **A 测量信号** → sample eval / full eval / scorer 单跑 vs majority 一致性 / 抽样规则代表性 / 工具链脚本（run_eval.sh / score.py / compare.py / extract_sample.py）的稳定性 / 跨 run 噪声量级
- **B 决策规则** → D1 tolerance 公式 / D2 终止条件 / EXIT PASS/REPLAN 阈值 / fix_plan 的 expected_tradeoffs / interaction_risks 假设是否被规则消费 / 核心 vs 基础 Iter 的差异化处理
- **C 跨层信号** → fix_plan ↔ D1 ↔ progressive_log ↔ 下轮 S1 的字段流转；prose-only 决策 rationale 下游能否机器消费；fix_plan 中"若 X 则 ROLLBACK"是否有可执行实现
- **D 成本曲线** → 本轮 HITL 介入次数 / 同一类问题跨轮重复出现 / baseline 跨轮漂移 / 同字段被多个组件重写 / ad-hoc 脚本反复手写

## 数值一致性必查
- baseline / threshold / tolerance / majority k——多处出现的同一概念数字对得齐吗
- 跨 round 比较口径一致（majority vs majority，不能 majority vs single-run）
- empty output 占比是否在容差内
